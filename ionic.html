    Cordova

    Advantages

    • Write HTML5 app, run on multiple platforms.
    • Only code the business logic ONCE.
    • Hard to tell from native code.

    Rainbows and unicorns?

    • When I first pursued this course, I encountered some resistance from clients and even devs. IMO Cordova has a bad rap because in its earlier incarnations these apps did not look as good. But mobile devices have improved nad software has improved.
    • These days in a lot of cases the average user can't even tell if an app uses Cordova or not (more so on Android).

    Cordova

    Challenges

    • You still have to understand your platform.
    • You still might have to write native code, depending on whether there's already a plugin.
    • Different platforms - different browsers.

    Cordova

    Plugin Architecture

    • Cordova by itself is not a complete solution.
    • Plugins expose the native api of the mobile device.
    • Examples: current location, camera, notifications, splash screen, open link in native browser
    • PhoneGap is a plugin!
    • Plugins Registry: http://plugins.cordova.io/
    • Cordova gives you the development framework etc. It does not give you an interface to native functions.

    Cordova: Is it PhoneGap?

    A: Sorta

    • I'd like to get this one out of the way first. When I first decided to write a multiplatform mobile app, I investigated the tools.
    • I found conflicting answers.
    • Part of the reason for this is that before Cordova was called Cordova, it was called "PhoneGap"!
    • However, PhoneGap is built "on top of" Cordova, and offers some additional features, including the ability to do a "remote build" of iOS applications, an advantage for us Windows developers.
    • Cordova is open source, and PhoneGap is an Adobe product. At this time, Adobe distributes it for free.
    • So you'll hear a lot of people use them interchangeably, but this presentation is about Cordova without PhoneGap

    Cordova

    Project Structure

    • Native code under "platforms"
    • HTML code under "www"
    • all in all, pretty familiar.
    • blow through this one. The point is that you are in familiar territory.
    • Why? HTML5!

    Running the Cordova App

    • In the browser!
    • Ripple for Android (not the chrome extension)
    • Emulators (for either platform)
    • Your phone.
      • Android - usb cable
      • iPhone - Add the id of your phone and Test Flight code to your app, sign up for Test Flight and set up an account, upload your app there, download it and you're all set!
    • advantages/disadvantages. For example, android emulator is very slow....
    • In the browser is very quick, but plugins don't work.
    • Chrome Canary has tools
    • After a lot of messing around, I use chrome (quick/layout) and the phone itself (the final authority).

    The Cordova Build

    • Runs in Ant
    • Hooks (i.e. build event scripts) can be added.
    • Build for one platform at a time, or multiple.
    • Ant is behind the scenes
    • sorry, Grunt/Gulp fans - the good news is that I have not had to change this at all. But good for understanding
    • Hooks - for injecting functionality into the build process. Usually node.js is recommended (x-platform).
      Can differentiate between build types. examples: diff icons or terms of use
    • Ant targets allow you to build, package, deploy
